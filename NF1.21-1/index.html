入门指南 (Getting Started)

入门NeoForge (Getting Started with NeoForge)

这一部分包含建立NeoForge工作区，运行并测试你的mod的方法。

前置条件 (Prerequisites)

  • 熟悉Java编程语言，特别是Java的面向对象、多态、泛型、函数式编程。
  • 安装Java 25开发工具包（Java 25 Development Kit, JDK）和64位Java虚拟机（64-bit Java Virtual Machine, JVM）。NeoForge官方推荐并支持微软版本的OpenJDK，但是其他JDK也可行。

⚠ 注意
请确保你在使用64位JVM。可以通过在终端执行java --version来检查。Minecraft不支持32位JVM。

  • 熟悉一种集成开发环境（Integrated Development Environment, IDE）。
    • NeoForge官方推荐IntelliJ IDEA和Eclipse，这两者都集成了对Gradle的支持。然而，可以使用任何IDE，诸如Netbeans、Visual Studio Code、Vim和Emacs。
  • 熟悉Git和GitHub。严格来说这是不必要的，但是会让你的开发过程简单得多。

建立工作区 (Setting Up the Workspace)

  • 前往Mod生成器网页，输入mod名称（以及可选地，输入mod id），包名，Minecraft版本和Gradle插件（ModDevGradle或者NeoGradle），点击“下载Mod项目”（”Download Mod Project”），并解压下载的ZIP文件。
  • 打开你的IDE并导入该Gradle项目。Eclipse和IntelliJ IDEA会自动为你完成。如果你的IDE不能完成，你也可以通过终端命令gradlew来完成。
    • 第一次执行时，Gradle会下载NeoForge的所有依赖（包括Minecraft自身）并反编译它们。这可能花费大量时间（或达到1小时左右，具体时间取决于你的硬件设备和网络状况）。
    • 每当你对Gradle文件进行修改，都需要重新加载Gradle更改，可以通过IDE的“重新加载Gradle”（”Reload Gradle”）按钮或者通过终端命令gradlew完成。

自定义你的Mod信息 (Customizing Your Mod Information)

你的Mod的许多属性都可以在gradle.properties文件中更改。这些属性包含mod名称和mod版本等。有关更多信息，请参阅gradle.properties文件中的注释，或者gradle.properties的文档。
如果你想要修改构建步骤，你可以编辑build.gradle和settings.gradle文件。NeoForge提供的Gradle插件（ModDevGradle或者NeoGradle）提供了若干配置选项，在构建脚本里的注释中解释了其中一些。

⚠ 注意
只有当你知道自己在做什么的时候才能去编辑build.gradle和settings.gradle文件。所有基本属性都可以通过gradle.properties来设置。

构建并测试你的Mod (Building and Testing Your Mod)

运行gradlew build来构建你的mod。这将会输出在build/libs中名为<archivesBaseName>-<version>.jar的文件。<archivesBaseName>与<version>分别是build.gradle中设置的属性，其默认值分别是gradle.properties文件中的mod_id和mod_version。如果需要，可以在build.gradle中修改命名。输出的JAR文件可以放置在启用NeoForge的Minecraft目录中的mods文件夹，或者上传到mod发布平台。
为了在测试环境中运行你的mod，你可以用IDE的运行配置或者相关的Gradle任务（例如gradlew runClient）。这将会从相应目录启动Minecraft（例如runs/client或者runs/server），同时包含任何指定的源代码集。默认的MDK包含了main源代码集，所以任何写在src/main/java中的代码都会运行。

服务端测试 (Server Testing)

如果你运行一个专用服务器，无论是通过IDE的运行配置还是gradlew runServer，这个服务器都会立刻关闭。你需要通过编辑运行目录中的eula.txt文件来接受Minecraft EULA。
一旦接受了EULA，服务器便会加载并在localhost（127.0.0.1）下运行。然而，你仍然不能加入服务器，因为服务器默认开启在线模式并进行身份验证（开发者玩家并不能通过正版验证）。为了修复，再次关闭你的服务器并将server.properties文件中的online-mode属性设置为false。现在，你应该可以连接你的服务器了。

💡 提示
你应该也在专用服务器环境中测试你的mod，包括仅客户端的mod，因为它们在服务器中加载时不应该做任何事。

Mod文件 (Mod Files)

Mod文件负责确定哪些mod会打包进你的JAR文件、那些信息会显示在“Mods”菜单中以及你的mod应该如何在游戏中加载。

gradle.properties (gradle.properties)

gradle.properties文件包含你的mod的各种常用的属性，例如mod id和mod版本。在构建时，Gradle读取文件中的这些值并将它们内联到不同位置，例如neoforge.mods.toml文件。这样，你只需要在一个地方更改属性值就可以将它们应用于任何地方。
MDK的gradle.properties文件的注释中解释了大多数属性。

属性 说明 举例
org.gradle.jvmargs 允许你传递额外的JVM参数。最常见的是将其用作给Gradle分配更多/更少内存。注意这只针对Gradle，不包括Minecraft。 org.gradle.jvmargs=-Xmx3G
org.gradle.daemon Gradle是否应该在构建时使用守护进程。 org.gradle.daemon=false
org.gradle.parallel Gradle是否应该派生多个JVM进程来并行执行多个项目。 org.gradle.parallel=false
org.gradle.caching Gradle是否应该重新使用先前构建任务的输出。 org.gradle.caching=false
org.gradle.configuration-cache Gradle是否应该是重新使用先前构建使用的构建配置。 org.gradle.configuration-cache=false
org.gradle.debug Gradle是否启用debug模式。Debug模式主要意味着Gradle日志有更丰富的输出。注意这只针对Gradle，不包括Minecraft。 org.gradle.debug=false
minecraft_version 你正在使用的Minecraft版本。必须与neo_version相匹配。 minecraft_version=1.20.6
minecraft_version_range 这个mod使用的Minecraft版本范围，采用Maven版本范围（Maven Version Range）格式。注意快照版本，预发布版与候选正式版不一定被正确排序，因为它们不符合maven版本格式。 minecraft_version_range=[1.20.6, 1.21)
neo_version 你正在使用的NeoForge版本。必须与minecraft_version匹配。有关更多信息，请参阅NeoForge的版本。 neo_version=20.6.62
mod_id 请参阅Mod ID。 mod_id=examplemod
mod_name 人类可读的mod显示名字。默认地，mod名称只能在mod列表中看见。然而，像JEI等mod会在物品提示中显示mod名称。 mod_name=Example Mod
mod_license Mod提供的许可证。推荐设置为你的许可证的SPDX标识，以及/或者你的许可证的链接。你可以访问https://choosealicense.com/来选择一个你想要的许可证。 mod_license=MIT
mod_version 你的mod的版本（将会在mod列表中显示）。有关更多信息，请参阅版本。 mod_version=1.0
mod_group_id 请参阅组ID。 mod_group_id=com.example.examplemod

Mod ID (The Mod ID)

Mod id是区分你的mod和其他mod的主要方式。它在各种地方都有广泛使用，例如，它被用作为mod注册表的命名空间。若两个mod拥有相同的id，游戏将不会被加载。
因此，你的mod id应该是独一无二且方便记忆的。Mod id只可能包含小写字母、数字和下划线，并且长度在2个和64个字符之间。

ℹ 注意
gradle.properties文件中的属性更改会自动应用在任何地方，除了你的mod主类中的@Mod注解。你需要手动匹配那里的值。

组ID (The Group ID)

正确设置build.gradle中的group属性是一个好习惯，虽然只有你打算在maven发布你的mod时它才是必要的。这是通过修改gradle.properties中的mod_group_id属性来完成的。
组ID应该设置为你的顶级包名。有关更多信息，请参阅包。

src/main/java中的包现在也应该符合这个结构，并用一个子包来代表mod id。

neoforge.mods.toml (neoforge.mods.toml)

src/main/resources/META-INF/neoforge.mods.toml文件是TOML格式的，定义了你的mod的元数据，也包含了你的mod应该怎样被游戏加载和“Mods”菜单中怎样显示你的mod信息。MDK提供的neoforge.mods.toml文件中的注释解释了每个条目，在这里将更详细地解释它们。
neoforge.mods.toml可以被分为三个部分：非特定于Mod的属性（与mod文件绑定）、mod属性（包含每个mod）和依赖配置（包含每一个mod的依赖）。一些neoforge.mods.toml是必要的，它们需要一个特定的值，否则会产生错误。

ℹ 注意
在默认的MDK中，Gradle将这个文件中的不同属性都替换为gradle.properties文件中指定的属性。例如，license="${mod_license}"意味着用gradle.properties中的mod_license属性来替换license字段。被替换的值应该在gradle.properties中更改，而不是在neoforge.mods.toml中。

非特定于Mod的属性 (Non-Mod-Specific Properties)

非特定于mod的属性是与JAR文件本身相关的属性，说明了怎么加载mod和附加的全局元数据。

属性 类型 默认 说明 举例
modLoader string javafml Mod使用的语言加载器。可以用来支持其他可选的语言结构（例如使用Kotlin的主文件）和mod入口点的不同形式（例如一个接口或者一个方法）。NeoForge提供了Java加载器javafml。 modLoader="javafml"
loaderVersion string "" 可接受的语言加载器版本范围，用Maven版本范围（Maven Version Range）表示。对于javafml，这是当前的版本1。如果没有指定版本，则任何版本的mod加载器都可以使用。 loaderVersion="[1,)"
license string 必要的 JAR中mod提供的许可证。推荐设置为你的许可证的SPDX标识，以及/或者你的许可证的链接。你可以访问https://choosealicense.com/来选择一个你想要的许可证。 license="MIT"
showAsResourcePack boolean false 当设置为true时，mod的资源将会作为一个独立的资源包（显示在“资源包”菜单），而不是与其他mod的资源结合成“Mod Resources”资源包。 showAsResourcePack=true
showAsDataPack boolean false 当设置为true时，mod的数据文件将会作为一个独立的数据包（显示在“数据包”菜单），而不是与其他mod的资源结合成“Mod Data”数据包。 showAsDataPack=true
services array [] 一个包含mod使用的服务的数组。这将纳入NeoForge基于Java平台模块系统（Java Platform Module System）为mod创建的模块的。 services=["net.neoforged.neoforgespi.language.IModLanguageProvider"]
properties table {} 一个包含替换属性的表格。它被StringSubstitutor用来替换${file.<key>}为相应的值。 properties={"example"="1.2.3"}（可以通过${file.example}来引用）
issueTrackerURL string 空 一个代表了报告mod、追踪mod问题的URL。 issueTrackerURL="https://github.com/neoforged/NeoForge/issues"

ℹ 注意
services属性在功能上等效于指定模块中的uses属性，用于允许加载给定类型的服务。
或者，它可以在src/main/resources/META-INF/services文件夹中的服务文件中定义，这些文件的文件名是完全规范的服务名称，文件内容是要加载的服务（请参阅AtlasViewer mod提供的范例）。

特定于Mod的属性 (Mod-Specific Properties)

特定于mod的属性通过[[mods]]头与特定mod绑定。这是表数组，所有键/值属性都会附加在这个mod中，直至到下一个头。

属性 类型 默认 说明 举例
modId string 必要的 请参阅Mod ID。 modId="examplemod"
namespace string value of modId mod的命名空间。必须符合mod id的格式，但可以额外附加小数点(.)和连接号(-)。目前未被使用。 namespace="example"
version string "1" Mod的版本，最好使用Maven版本的变体。当设置为${file.jarVersion}，它将会被JAR清单文件中的Implementation-Version属性的值（在开发环境中显示为0.0NONE）替换。 version="1.20.2-1.0.0"
displayName string value of modId Mod的显示名称。在屏幕上代表mod（例如，在mod列表中和mod版本不匹配提示中显示）。 displayName="Example Mod"
description string '''MISSING DESCRIPTION''' Mod的描述，在mod列表屏幕中显示。推荐使用多行字符串。这个值也是可翻译的，有关更多信息，请参阅翻译Mod元数据。 description='''This is an example.'''
